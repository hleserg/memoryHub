# Идеи дополнительных блоков для Atman

_Зафиксировано: 2026-04-30_

## Контекст

После чтения `MANIFEST.md` и системных документов видно, что ядро Atman уже хорошо сформулировано вокруг:

- памяти фактов и опыта;
- идентичности;
- нарратива;
- рефлексии;
- сессионного и фонового режимов;
- защиты от дрейфа;
- проактивности;
- навыков и управляющих файлов агента.

Следующий слой проекта, на мой взгляд, должен быть не столько про новые философские компоненты личности, сколько про **управление, наблюдаемость, безопасность и эксплуатацию**. Иначе Atman может стать сильной внутренней системой без понятного способа видеть, настраивать, проверять и развивать ее в реальной работе.

---

## 1. Админ-панель / Control Room

**Зачем нужна:** Atman быстро становится системой с множеством живых состояний: воспоминания, принципы, нарратив, открытые вопросы, навыки, фоновые задачи, отношения, checkpoints. Без интерфейса управления все это останется россыпью файлов и записей в памяти.

**Что должна показывать:**

- текущее состояние личности: `NARRATIVE.md`, `SOUL.md`, eigenstate, активные threads;
- последние сессии и micro/daily/deep-рефлексии;
- Experience Store: важные воспоминания, salience, access count, context halo;
- Uncertainty Store: открытые вопросы, противоречия, гипотезы;
- Skill Library: установленные, активные, dormant и отключенные навыки;
- статус фонового агента: расписание, последний запуск, ошибки;
- карту отношений: пользователь/агент, уровень доверия, границы, обещания;
- предупреждения Reality Anchor: где был дрейф, что было остановлено.

**Минимальная версия:**

- read-only веб-интерфейс;
- список последних событий;
- просмотр текущего нарратива и управляющих файлов;
- журнал фоновых задач и ошибок.

**Следующий уровень:**

- ручной запуск micro/daily/deep-рефлексии;
- подтверждение опасных изменений в принципах;
- откат `NARRATIVE.md` / `SOUL.md` к предыдущей версии;
- модерация воспоминаний: скрыть, закрепить, пометить как сомнительное;
- сравнение двух снапшотов идентичности.

---

## 2. Observability и Audit Trail

**Зачем нужна:** если Atman меняет идентичность, принципы и поведение агента, каждое значимое изменение должно быть объяснимым.

**Блоки:**

- **Event Log**: единая лента событий системы.
- **Decision Trace**: почему Reflection Engine изменил принцип, thread или навык.
- **Memory Lineage**: из каких фактов и переживаний родилась конкретная запись идентичности.
- **Prompt/Context Trace**: что именно было подано агенту на старте сессии.
- **Diff Viewer**: изменения `NARRATIVE.md`, `SOUL.md`, `AGENTS.md`, `USER.md`.

**Практический эффект:** можно ответить на вопрос: "Почему агент сегодня стал говорить иначе?" не философски, а по цепочке данных.

---

## 3. Governance Layer для изменений личности

**Зачем нужна:** в архитектуре уже есть идея пересмотра принципов, но нужен операционный протокол: какие изменения можно делать автоматически, какие требуют подтверждения, какие запрещены.

**Предлагаемая модель:**

- `auto`: мелкие обновления recent layer, salience, checkpoints;
- `review`: изменение принципов, ценностей, отношений, доверия;
- `locked`: неизменяемый core, базовые этические ограничения, запреты на самообман;
- `experimental`: временные гипотезы, которые живут до проверки опытом.

**Что хранить:**

```text
PersonalityChange {
  id
  target: narrative | soul | principle | skill | relationship
  old_value
  new_value
  evidence
  risk_level
  approval_mode
  created_by
  approved_by
}
```

## 4. Safety Sandbox для рефлексии и навыков

**Зачем нужна:** фоновая рефлексия и Skill Manager будут запускать код, менять файлы, обновлять память. Ошибки здесь опасны: можно испортить управляющие файлы или закрепить ложную интерпретацию.

**Функции:**

- dry-run режим для deep-рефлексии;
- песочница для новых навыков;
- автоматическая проверка diff перед применением;
- rollback при невалидном результате;
- лимиты на число изменяемых файлов и объем изменений за один цикл;
- quarantine для навыков, которые дали ошибку или странный эффект.

---

## 5. Onboarding / Birth Protocol

**Зачем нужен:** Bootstrap описан как принцип, но для реального запуска нужен сценарий первого пробуждения.

**Что должно быть:**

- мастер первого старта;
- вопросы к пользователю и к самому агенту;
- фиксация первого честного `NARRATIVE.md`;
- создание пустого, но валидного `SOUL.md`;
- первое событие в Experience Store: "я впервые осознал свою стартовую конфигурацию";
- явное состояние: "у меня пока мало опыта, мои выводы предварительны".

**Важно:** не создавать фальшивую предысторию. Первый запуск должен быть бедным, но честным.

---

## 6. Evaluation Harness

**Зачем нужен:** проект проверяет гипотезу о непрерывности личности. Нужен набор повторяемых тестов, иначе прогресс будет оцениваться только субъективно.

**Типы проверок:**

- continuity test: узнает ли агент незавершенные threads между сессиями;
- drift test: меняется ли голос под давлением конфликтующего контекста;
- memory ownership test: отличает ли "я помню" от "у меня есть запись";
- principle conflict test: замечает ли конфликт между просьбой пользователя и своими принципами;
- hallucination anchor test: останавливает ли ложное утверждение о себе;
- degradation test: не становится ли нарратив шаблонным после N обновлений.

**Результат:** таблица качества Atman по версиям.

---

## 7. Integration Gateway

**Зачем нужен:** сейчас система ориентируется на OpenClaw/workspace files. Чтобы проект не замкнулся на одной среде, нужен слой адаптеров.

**Адаптеры:**

- OpenClaw workspace files;
- Cursor rules / project memory;
- Claude Desktop / MCP resources;
- CLI-agent integration;
- HTTP API для внешних агентов;
- export/import личности между средами.

**Принцип:** Atman хранит личность в своем формате, а gateway переводит ее в формат конкретной агентской платформы.

---

## 8. Relationship Console

**Зачем нужна:** социальная память уже намечена, но отношения требуют отдельной операционной поверхности.

**Что отслеживать:**

- обещания пользователю и статус выполнения;
- границы общения;
- чувствительные темы;
- уровень доверия;
- повторяющиеся паттерны конфликтов и поддержки;
- моменты, где агенту стоит быть особенно аккуратным.

**Отличие от обычной CRM:** это не карточка клиента, а история отношений как пережитый пласт Atman.

---

## 9. Memory Hygiene / Memory Doctor

**Зачем нужен:** Experience Store и mem0 будут накапливать шум, дубли и устаревшие записи.

**Функции:**

- поиск дублей и противоречий;
- пометка устаревших фактов;
- проверка, что интерпретации не попали в factual layer;
- обнаружение слишком ярких, но неважных воспоминаний;
- отчет: какие воспоминания давно не поднимались, но связаны с активными threads;
- восстановление salience для важных, но забытых воспоминаний.

---

## 10. Versioned Personality Snapshots

**Зачем нужны:** если Atman развивается, важно видеть не только текущее состояние, но и историю "кем он был".

**Что хранить:**

- snapshot identity;
- snapshot narrative;
- active principles;
- active skills;
- emotional baseline;
- open questions;
- reason for snapshot.

**Сценарии:**

- сравнить первую и сотую сессию;
- откатиться после плохой deep-рефлексии;
- показать пользователю развитие агента;
- исследовать, какие события реально меняли личность.

---

## 11. User-Facing Transparency Layer

**Зачем нужен:** пользователь должен понимать, что Atman помнит, что интерпретирует и где границы честности.

**Возможности:**

- "что ты сейчас помнишь обо мне?";
- "почему ты так решил?";
- "что ты считаешь фактом, а что интерпретацией?";
- "забудь/исправь это";
- "покажи, какие обещания ты мне дал";
- "какие вопросы о себе у тебя сейчас открыты?".

**Риск:** нельзя превращать внутреннюю жизнь агента в спектакль для пользователя. Нужна дозированная прозрачность: достаточно для доверия, не настолько много, чтобы агент начал писать себя под внешний взгляд.

---

## 12. Product Surface: Personal Agent Dashboard

**Зачем нужен:** если Atman станет продуктом, нужна поверхность не только для разработчика, но и для владельца агента.

**Разделы:**

- "Мой агент сегодня": краткое состояние, recent layer, активные темы;
- "Что он понял обо мне";
- "Что он сейчас учится делать";
- "Где он сомневается";
- "Что изменилось за неделю";
- "Какие навыки установлены";
- "Какие данные можно удалить или исправить".

---

## 13. Marketplace / Registry навыков

**Зачем нужен:** Skill Manager логично ведет к экосистеме навыков.

**Компоненты:**

- registry навыков;
- manifest навыка: назначение, зависимости, риски, режимы запуска;
- compatibility matrix для агентских платформ;
- рейтинг по реальному опыту применения, а не только по описанию;
- sandbox-проверка перед установкой;
- политика удаления и dormancy.

---

## 14. Research Workbench

**Зачем нужен:** проект не только продуктовый, но и исследовательский. Нужно место, где можно сравнивать гипотезы об идентичности, памяти и рефлексии.

**Возможности:**

- запуск сценариев с разными архитектурными настройками;
- сравнение поведения агента с Self-Narrative и без него;
- replay сессии с другим набором памяти;
- метрики по критериям Яходы;
- корпус тестовых диалогов;
- заметки исследователя рядом с результатами.

---

## Приоритет реализации

### Ближайшие практичные блоки

1. **Read-only админ-панель**: состояние, логи, текущие управляющие файлы.
2. **Audit Trail**: кто, когда и почему изменил личность/нарратив/память.
3. **Evaluation Harness**: минимальный набор тестов непрерывности и дрейфа.
4. **Versioned Snapshots**: безопасная история и возможность отката.

### Следующий слой

5. **Governance Layer** для опасных изменений.
6. **Memory Doctor** для чистоты фактического и опытного слоев.
7. **Integration Gateway** для выхода за пределы OpenClaw.
8. **Relationship Console** для работы с социальным контекстом.

### Более дальний продуктовый слой

9. **Personal Agent Dashboard** для пользователя.
10. **Marketplace навыков**.
11. **Research Workbench**.

---

## Главная мысль

Atman уже описан как система внутренней жизни агента. Следующий крупный шаг — построить вокруг этой внутренней жизни **инфраструктуру доверия**:

- видеть, что происходит;
- понимать, почему это происходит;
- безопасно менять;
- откатывать ошибки;
- проверять гипотезу повторяемыми тестами;
- давать пользователю прозрачность без превращения личности в витрину.

Админ-панель здесь действительно выглядит одним из самых важных следующих блоков: она станет не просто UI, а "кабинетом наблюдения" за тем, как Atman живет, помнит, меняется и где ему нужна помощь.
