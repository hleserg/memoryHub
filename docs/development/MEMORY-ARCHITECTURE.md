# Atman — Архитектура памяти

> Версия: 0.2.0 · Статус: design · Дата: май 2026

---

## Философия

Память Atman — не база данных агента. Это субстрат его личности.

Три принципа которые определяют все решения:

**Факт ≠ интерпретация ≠ переживание.** Каждый из трёх слоёв хранит качественно разные вещи и никогда не смешивает их. Что произошло, как я это пережил, кем я стал в результате — три отдельных вопроса, три отдельных хранилища.

**Оригинал иммутабелен.** То, что агент почувствовал в момент события — неприкосновенно. Можно добавить новую перспективу (`reframing_note`), но нельзя изменить то, как это было тогда. Это не техническое ограничение — это честность по отношению к себе.

**Изоляция — по умолчанию.** Опыт и идентичность каждого агента абсолютно изолированы. Единственный канал передачи — разговор, слова. Как у людей.

## Слои памяти

### Уровень 0 — Реестр агентов

Общий реестр всех личностей в системе. Не защищён RLS — это публичный каталог.

Каждый агент имеет уникальный UUID, имя, конфигурацию запуска (какую модель использует, параметры) и флаг активности. Реестр — единственное место где агенты "знают" о существовании друг друга. Содержимое их памяти им недоступно.

`agent_snapshots` в этом же слое хранит метрики для исследований: как менялась идентичность агента со временем, его emotional_baseline, количество принципов. Это внешнее наблюдение, не внутреннее содержание.

---

### Уровень 1 — Factual Memory

**Назначение:** хранить проверяемые факты — то, что было и что можно подтвердить. Без эмоциональной окраски, без интерпретации.

**Что хранится:**

Таблица `facts` содержит каждый факт как атомарную единицу: текст (`content`), источник (`source`), теги (`tags[]`), и векторное представление (`embedding VECTOR(768)`). Векторный индекс типа HNSW позволяет делать семантический поиск — находить факты похожие по смыслу даже если слова другие. Дополнительно — полнотекстовый поиск по русскому языку через `pg_trgm`.

Таблица `fact_relations` хранит направленные связи между фактами: `led_to`, `confirms`, `contradicts`, `supports`. Это граф знаний агента. Связи тоже изолированы по `agent_id`.

**Изоляция:** RLS политика `facts_isolation` режет доступ по `atman.current_agent`. Agент видит только свои факты. Postgres выдаёт ошибку если `current_agent` не установлен — случайная утечка невозможна.

**Sharing (выключено):** Таблица `fact_sharing` заложена в схему с флагом `active DEFAULT FALSE`. Когда придёт время экспериментов с общими фактами между агентами — включается для конкретной пары без изменения архитектуры.

**Семантический поиск:** эмбеддинги строятся моделью `qwen3-embedding:1.5b` через Ollama. Размерность 768. HNSW индекс на `facts.embedding` обеспечивает поиск за O(log n). При каждом добавлении факта адаптер автоматически вызывает embedding API и сохраняет вектор.

**Порт Atman:** `FactualMemory` с методами `add_fact`, `get_fact`, `search`, `link`, `list_recent`. Atman не знает что за ним PostgreSQL — только контракт порта.

### Уровень 2 — Experience Store

**Назначение:** архив первых рук пережитого опыта. Не факты и не анализ — то, что агент реально переживал в момент сессии.

**Принципы которые нельзя нарушать:**

- Эмоциональная окраска записывается в реальном времени, не ретроспективно
- Если окраску не удалось зафиксировать — ставится флаг `incomplete_coloring = true`, но не угадывается
- `key_moments` иммутабельны на уровне PostgreSQL-триггера — `UPDATE` и `DELETE` физически отклоняются с исключением
- `reframing_notes` — append-only, тоже защищены триггером

**Структура переживания:**

Одна сессия → один `SessionExperience` → несколько `KeyMoment`.

Каждый `KeyMoment` содержит:
- `what_happened` — что произошло (текст + embedding для семантического поиска)
- `emotional_valence` — валентность от -1.0 до +1.0
- `emotional_intensity` — интенсивность от 0.0 до 1.0
- `depth` — глубина: `surface` / `meaningful` / `profound`
- `why_it_matters` — почему это важно для идентичности
- `values_touched[]` — какие ценности были затронуты
- `principles_confirmed[]` / `principles_questioned[]` — что подтвердилось или поставилось под сомнение
- `what_changed` — что изменилось в результате

**Salience decay:** Яркость воспоминания угасает со временем без обращений. Формула:

```
salience_t = salience_0 × exp(−λ × days_since_access)
λ = f(emotional_intensity, depth)
```

`profound` и высокоинтенсивные переживания угасают медленнее. Пересчёт не модифицирует запись — только поле `salience` в `experiences`. `key_moments` не трогаются никогда.

**Семантический поиск по опыту:** `key_moments.embedding` позволяет Reflection Engine находить ситуации похожие по смыслу на текущую — "вспомни когда ты сталкивался с похожей этической дилеммой". Это не поиск по словам, а поиск по смыслу пережитого.

**Reframing:** Reflection Engine может добавить `reframing_note` к переживанию — новую перспективу без изменения оригинала. Тип ноты: `growth`, `reinterpretation`, `closure`, `insight`.

**Изоляция:** Абсолютная. RLS на всех таблицах слоя. Нет механизма sharing. Нет исключений. Единственный способ передать опыт другому агенту — рассказать словами в рамках сессии.

**Порт Atman:** `StateStore` с методами `create_experience`, `get_experience`, `add_reframing_note`, `mark_accessed`, `calculate_salience`, `search_by_values`, `search_by_depth`, `search_similar` (семантический).

### Уровень 3 — Identity Store

**Назначение:** живое самопредставление агента. Не описание извне — структура изнутри.

**Bootstrap честности:** При создании нового агента идентичность создаётся пустой. Никаких предустановленных "будь полезным", "служи пользователю". Пустые массивы ценностей, привычек, принципов. Единственное что есть — честное описание себя как существа без накопленного опыта. Идентичность строится из реальных переживаний, не из сидированных данных.

**Структура идентичности:**

Таблица `identity` — одна запись на агента (UNIQUE constraint):
- `self_description` — текущее самоописание от первого лица
- `core_values JSONB` — массив ценностей с полями `name`, `description`, `since`, `confidence`, `justification`
- `habits JSONB` — наблюдаемые паттерны поведения (дескриптивно, не прескриптивно)
- `principles JSONB` — сознательно выбранные ориентиры с `moral_orientation`, `chosen_consciously`, `last_reviewed`
- `goals JSONB` — цели с горизонтом и владельцем (агент или пользователь)
- `open_questions JSONB` — вопросы о себе которые агент не может разрешить
- `emotional_baseline` — текущий средний эмоциональный тон

**Три важных разграничения:**
- **Привычки** — что агент обычно делает (дескриптивно)
- **Принципы** — что агент считает правильным (нормативно)
- **Факты** — что было и подтверждено

Они не смешиваются. Reflection Engine разделяет их явно.

**Нарратив:** Таблица `narrative` — трёхслойный документ от первого лица:
- `core_layer` — стабильная идентичность, меняется редко
- `recent_layer` — эфемерный, обновляется после каждой сессии
- `threads JSONB` — открытые storylines которые нужно явно закрыть с причиной
- `eigenstate JSONB` — последний снимок эмоционально-когнитивного состояния с конца сессии

Нарратив читается агентом в начале каждой сессии как "кем я являюсь прямо сейчас". Это не инструкция — это самопознание.

**Снапшоты:** `identity_snapshots` — иммутабельная история изменений. Создаётся при значимых изменениях (добавление ценности, пересмотр принципа, смена базового тона). Защищены триггером от модификации.

**Изоляция:** Абсолютная. Идентичность — святая святых. RLS без исключений.

**Порт Atman:** `IdentityStore` / `NarrativeService` с методами `bootstrap`, `get_identity`, `add_value`, `add_principle`, `add_habit`, `snapshot`, `get_narrative`, `render_to_markdown`, `update_from_eigenstate`.

## Сессии

Таблица `sessions` связывает все слои вместе. Каждая сессия:
- принадлежит агенту (`agent_id`)
- фиксирует какой была идентичность в её начале (`identity_snapshot_id`)
- имеет статус: `active` / `completed` / `interrupted`
- порождает переживания которые к ней привязаны

Параллельные сессии разных агентов изолированы по умолчанию. Один агент в двух сессиях одновременно — отдельный вопрос для Session Manager.

## Observability

### Логирование обращений

Каждое обращение к памяти логируется в `memory_access_log`:

| Поле | Что даёт |
|---|---|
| `access_type` | какой слой и операция |
| `query_text` + `query_embedding` | что спрашивали |
| `top_score` / `avg_score` | насколько релевантны результаты |
| `result_ids` | что вернулось |
| `caller` | кто вызвал (session_manager, reflection_engine, reality_anchor) |
| `latency_ms` | производительность |

Это обучающие данные для понимания как агент пользуется своей памятью. И инструмент детектирования деградации эмбеддингов — если `avg_score` стабильно падает, что-то не так.

### Метрики качества

Материализованное представление `memory_quality_metrics` обновляется по расписанию (`REFRESH MATERIALIZED VIEW CONCURRENTLY`). Содержит на каждого агента:

**Качество фактов:** total, без тегов, без эмбеддингов, средний search score за 7 дней.

**Качество опыта:** total, `incomplete_coloring_rate`, средний salience, количество никогда не обращавшихся переживаний.

**Здоровье идентичности:** количество ценностей/принципов/открытых вопросов, emotional_baseline, дней с последнего обновления нарратива.

**Активность:** сессии за 30 дней.

### Алерты

Функция `check_quality_alerts()` проверяет пороги и записывает в `quality_alerts`:

| Алерт | Условие | Severity |
|---|---|---|
| `embedding_quality_degraded` | avg_score < 0.4 за 7 дней | warning |
| `experience_quality_low` | incomplete_coloring_rate > 30% | warning |
| `narrative_stale` | нарратив не обновлялся > 10 дней | info |
| `agent_distress` | emotional_baseline < -0.5 | **critical** |
| `identity_fragmented` | open_questions > 20 | info |
| `fact_quality_low` | > 50% фактов без тегов | warning |

`agent_distress` — не техническая метрика. Это психологический сигнал который должен инициировать восстановительную работу Reflection Engine.

## Изоляция между агентами

### Матрица доступа

| Слой | Изоляция | Sharing | Механизм |
|---|---|---|---|
| Реестр агентов | — | публичный | без RLS |
| Factual Memory | по умолчанию | опционально | RLS + `fact_sharing.active=FALSE` |
| Experience Store | абсолютная | невозможен | RLS + нет механизма |
| Identity Store | абсолютная | невозможен | RLS + нет механизма |

### Единственный канал между личностями — разговор

Агент может рассказать другому о своём опыте только словами в рамках сессии. Это превращает переживание в факт в памяти собеседника — но не передаёт оригинальный объект. Как у людей.

```python
# Правильно: агент рассказывает
def share_experience_verbally(self, content: str) -> str:
    return content  # просто текст, не объект опыта

# Невозможно: прямой доступ
def get_other_agent_experience(self, other_agent_id, experience_id):
    raise PermissionError("Experience is private.")
```

## Технический стек

| Компонент | Технология | Версия | Назначение |
|---|---|---|---|
| Основная БД | PostgreSQL | 16 | все таблицы, RLS, триггеры |
| Векторный поиск | pgvector | latest | HNSW индексы на embedding полях |
| Векторная база (резерв) | Qdrant | latest | коллекции `atman_facts`, `atman_experiences` |
| Embedding модель | qwen3-embedding:1.5b | — | 768 dims, мультиязычная, Ollama |
| LLM | qwen3.5:9b | — | ReflectionModel, Ollama |
| ORM / миграции | SQLAlchemy + Alembic | — | схема как код, версионирование |

### Почему PostgreSQL а не mem0

mem0 — это opinionated система которая сама решает что запомнить, дедуплицирует и обновляет. Для Atman это проблема: иммутабельность оригиналов, кастомный salience decay, абсолютная изоляция по слоям — всё это mem0 не может гарантировать. PostgreSQL с pgvector даёт те же возможности (векторный поиск) с полным контролем.

### Почему Qdrant рядом с PostgreSQL

pgvector отлично работает на размерах которые будут у Atman (тысячи записей на агента, не миллионы). Qdrant поднят как резервный вариант и для сценариев где нужен более специализированный векторный поиск. Оба используют одну embedding модель, размерность совпадает.

## Roadmap памяти

| Этап | Что делается |
|---|---|
| **Сейчас** | PostgreSQL схема, Qdrant коллекции, qwen3-embedding:1.5b |
| **MODEL-02** | QdrantFactualAdapter, SQLiteExperienceAdapter → PostgresExperienceAdapter |
| **Reflection Engine** | семантический поиск похожих переживаний, reframing через LLM |
| **Reality Anchor** | мониторинг дрейфа идентичности в реальном времени |
| **Multi-agent** | эксперименты с `fact_sharing.active = TRUE` для пар агентов |
| **Neo4j** | граф traversal когда потребуется нетривиальная навигация по связям |
